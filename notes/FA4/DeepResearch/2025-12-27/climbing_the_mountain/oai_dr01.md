## NVIDIA PTX ISA (v9.1, CUDA 13.1 – Dec 2025\)

**Link:** *Parallel Thread Execution ISA 9.1 (NVIDIA)* – [PDF](https://docs.nvidia.com/cuda/pdf/ptx_isa_9.1.pdf)

### Relevant sections

#### *Bulk-Asynchronous Copy (Tensor Memory Accelerator)*

cp.async.bulk.tensor is a **non-blocking** instruction initiating an asynchronous copy from a source to a destination state space. This variant uses an **mbarrier-based completion** mechanism: upon finishing the copy, a “complete-tx” operation is performed on the specified memory barrier (mbarrier) with a count equal to the number of bytes transferred[\[1\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=%60cp.async.bulk.tensor%60%20is%20a%20non,state%20space). In effect, the mbarrier’s transaction count is decremented as each async copy completes, and the operation has release semantics at cluster scope[\[2\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=The%20copy%20operation%20in%20,in%20the%20Memory%20Consistency%20Model). The alternative .bulk\_group modifier uses a bulk async-group completion mechanism (group-based waiting) instead[\[3\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=The%20modifier%20,group%20based%20completion%20mechanism).

**Key points:** \- **Non-blocking DMA:** cp.async.bulk.tensor offloads a multi-dimensional copy (1D–5D tile) to the Tensor Memory Accelerator (TMA) hardware without stalling the thread[\[1\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=%60cp.async.bulk.tensor%60%20is%20a%20non,state%20space). The thread can continue execution while data moves. \- **MBarrier completion:** Each TMA copy can signal completion by updating an **mbarrier** object. The PTX .mbarrier::complete\_tx::bytes qualifier means the hardware will issue an mbarrier.complete\_tx for the given byte count on the provided barrier when the transfer finishes[\[4\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=,amount%20of%20data%20copied%20in)[\[5\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=cp.async.bulk.tensor.1d.shared%3A%3Acta.global.mbarrier%3A%3Acomplete_tx%3A%3Abytes.tile%20%5BsMem0%5D%2C%20%5BtensorMap0%2C%20). \- **Bulk-group mode:** Optionally, multiple async copies can be grouped. Using the .bulk\_group modifier indicates an async-group completion model (threads will later call a collective wait on the group rather than per-copy barrier)[\[3\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=The%20modifier%20,group%20based%20completion%20mechanism). \- **Memory ordering:** The copy is a weak memory operation; the completion (mbarrier update) serves as a release operation to make the data visible to consumers at cluster or CTA scope as appropriate[\[2\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=The%20copy%20operation%20in%20,in%20the%20Memory%20Consistency%20Model).

#### *Memory Barrier (mbarrier) Semantics for Async Ops*

An **mbarrier object** is an opaque object in shared memory with `.b64` type and an alignment of **8 bytes**, used to synchronize threads and track async operations[\[6\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=An%20mbarrier%20object%20is%20an,initialized%20and%20invalidated%20using)[\[7\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=An%20mbarrier%20object%20is%20used,threads%20and%20asynchronous%20memory%20operations). (The “64-byte aligned” requirement applies to **CUtensorMap** descriptors, not mbarrier.) Each mbarrier maintains a **phase**, an **arrival count**, and (on Hopper and beyond) a **transaction count (tx-count)** for async copies. The *phase* increments each time the barrier is reused for a new cycle of synchronization, and the hardware automatically resets the barrier for the next phase once the current phase completes[\[8\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=Starting%20with%20the%20Hopper%20architecture,and%20yet%20to%20be%20complete)[\[9\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=,pending%20arrivals%20has%20reached%20zero). A phase is considered complete when **(a)** the pending thread arrivals count reaches zero *and* **(b)** the async tx-count reaches zero[\[9\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=,pending%20arrivals%20has%20reached%20zero). At that point, the mbarrier transitions to the next phase and can be reused without re-initialization[\[10\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=An%20mbarrier%20object%20is%20automatically,all%20prior%20phases%20are%20complete)[\[11\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=Phase%20transition).

**Key mechanisms:** \- **Arrive and expect:** When initializing an mbarrier, a thread specifies an expected number of arrivals (thread participants). Each mbarrier.arrive or …arrive\_barrier call decrements the pending arrivals count[\[12\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=An%20arrive,the%20following%202%20steps). For async ops, mbarrier.expect\_tx can be used to increment the **tx-count** by a specified amount (e.g. bytes of data to copy)[\[13\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=The%20expect,completion%20of%20additional%20asynchronous%20transactions). Typically a single thread sets the tx-count to the total bytes (or units) of asynchronous work to be tracked in this phase. \- **Complete-tx:** As each asynchronous transaction finishes, the hardware issues an mbarrier.complete\_tx on the barrier, decrementing the tx-count by the given amount[\[14\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=The%20tx,completion%20of%20the%20current%20phase)[\[15\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=The%20complete,object%20consists%20of%20the%20following). When the tx-count hits zero (and all thread arrivals have occurred), the barrier’s phase is considered satisfied[\[9\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=,pending%20arrivals%20has%20reached%20zero). \- **Phase auto-reset:** Upon phase completion, the barrier automatically transitions to the next phase: the phase counter increments, the pending arrivals count is reset to the originally expected count, and the tx-count resets to 0[\[11\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=Phase%20transition)[\[16\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=,to%20the%20next%20phase). This allows reusing the same mbarrier for a new round of async operations without an explicit re-init. \- **Initialization & scope:** mbarrier.init.b64 \[addr\], count; initializes a barrier at memory address addr with an expected arrivals count (1–2\<sup\>20\</sup\>-1). It sets phase=0, expected=count, pending=count, tx-count=0[\[17\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=)[\[18\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=,to%200). Barriers can reside in .shared::cta or .shared::cluster memory (for CTA-local or cluster-wide use), but not all operations are supported at cluster scope (e.g. some thread wait operations are CTA-only)[\[19\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=The%20following%20table%20summarizes%20the,at%20different%20shared%20memory%20locations)[\[20\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=Supported).

#### *tcgen05 Instructions (5th-Gen TensorCore Ops – Blackwell)*

**TensorCore Generation 5 (SM100+)** introduces new instructions prefixed tcgen05.\* for advanced async data and compute pipelines. These include asynchronous LD/ST to the **Tensor Memory**, synchronization primitives, and 5th-gen MMA ops. For example, **tcgen05.ld/tcgen05.st** perform tensor memory loads and stores, and **tcgen05.cp** initiates copies via the TMA unit (these complement the cp.async.bulk instructions at the PTX level)[\[21\]](https://docs.nvidia.com/cuda/parallel-thread-execution/contents.html#:~:text=match%20at%20L599%20,wait)[\[22\]](https://docs.nvidia.com/cuda/parallel-thread-execution/contents.html#:~:text=,5th%20Generation%20Matrix%20Multiply%20and). A notable addition is **tcgen05.wait**, which serves as a fine-grained barrier to wait for asynchronous operations (including TMA transactions or pipeline stages) to complete without using mbarriers[\[23\]](https://docs.nvidia.com/cuda/parallel-thread-execution/contents.html#:~:text=,wait%20instruction%20based%20completion)[\[24\]](https://docs.nvidia.com/cuda/parallel-thread-execution/contents.html#:~:text=,wait). These instructions form the building blocks of Blackwell’s *warp-specialized* pipelines, where threads in a warp (or warpgroup) can produce and consume data asynchronously.

In PTX ISA 9.1, the **tcgen05\* section** defines a pipelining model for these ops. There are two completion mechanisms for pipelined tcgen05 operations: an **mbarrier-based** scheme and a **tcgen05.wait\-based** scheme[\[23\]](https://docs.nvidia.com/cuda/parallel-thread-execution/contents.html#:~:text=,wait%20instruction%20based%20completion). The latter implies using a direct instruction (…wait) to block until a prior tcgen05 async op finishes, rather than relying on an mbarrier object[\[23\]](https://docs.nvidia.com/cuda/parallel-thread-execution/contents.html#:~:text=,wait%20instruction%20based%20completion)[\[25\]](https://docs.nvidia.com/cuda/parallel-thread-execution/contents.html#:~:text=tcgen05,Packing%20and%20Unpacking). For example, a tcgen05.mma\_async (asynchronous matrix multiply-accumulate) can be followed by tcgen05.commit\_group and tcgen05.wait\_group to group multiple MMA operations and then wait for their completion in hardware[\[26\]](https://docs.nvidia.com/cuda/parallel-thread-execution/contents.html#:~:text=wgmma.mma_async.m64nNk16%20%20,Accumulate%20Instruction)[\[27\]](https://docs.nvidia.com/cuda/parallel-thread-execution/contents.html#:~:text=%2A%209.7.15.7.1.%20Asynchronous%20Multiply,Accumulate%20Instruction%3A%20wgmma.wait_group). This reduces software overhead in coordinating complex pipelines.

**Key points:** \- **Async tensor ops:** tcgen05.ld/st/cp perform memory operations through the Tensor Memory fabric (TMEM). They allow asynchronous loading of tensor tiles and often pair with tcgen05.wait or barriers to ensure data is ready. \- **Pipeline control:** tcgen05.commit\_group and tcgen05.wait\_group are used to mark a group of asynchronous operations and then wait for the group’s completion[\[27\]](https://docs.nvidia.com/cuda/parallel-thread-execution/contents.html#:~:text=%2A%209.7.15.7.1.%20Asynchronous%20Multiply,Accumulate%20Instruction%3A%20wgmma.wait_group). This is analogous to the older …commit\_group/…wait\_group for cp.async and wgmma, but generalized for 5th-gen pipelines. \- **New MMA forms:** Fifth-gen Tensor Cores support larger “warpgroup-level” MMA instructions (tcgen05.mma\*) that span multiple warps (see below). These have variants for structured sparsity (tcgen05.mma.sp) and weight-streaming (tcgen05.mma.ws) with corresponding coordination instructions[\[28\]](https://docs.nvidia.com/cuda/parallel-thread-execution/contents.html#:~:text=tcgen05.mma%20,TensorCore%205th%20Generation%20Instructions). \- **Allocation and permits:** Additional ops like tcgen05.alloc, tcgen05.relinquish\_alloc\_permit manage tensor map allocation permits for TMEM, ensuring that threads coordinate memory tile usage[\[29\]](https://docs.nvidia.com/cuda/parallel-thread-execution/contents.html#:~:text=match%20at%20L595%20tcgen05,Packing%20and%20Unpacking).

#### *Warpgroup Matrix Multiply (wgmma) – Hopper vs. Blackwell*

**Warpgroup MMA (WGMM)** is the Hopper-era (SM90) feature allowing a group of warps (a *warpgroup*) to collaboratively execute a single large matrix multiply. A **warpgroup** is defined as a set of 4 contiguous warps (128 threads total) where the first warp’s index is a multiple of 4[\[30\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=,397)[\[31\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=A%20warpgroup%20is%20a%20set,is%20a%20multiple%20of%204). All warps in a warpgroup operate in lockstep on one wgmma.mma\_async instruction, producing a larger tile of the result. This was introduced with the wgmma.mma\_async PTX instruction (for FP16, BF16, TF32 and INT8 matrix multiplication shapes up to 64×256×K) on Hopper, along with **wgmma** proxy ops for synchronization. For example, after issuing asynchronous wgmma operations, threads use wgmma.fence to ensure all previous wgmma operations are visible, wgmma.commit\_group to mark a bundle of operations as a group, and wgmma.wait\_group to stall until the group completes[\[27\]](https://docs.nvidia.com/cuda/parallel-thread-execution/contents.html#:~:text=%2A%209.7.15.7.1.%20Asynchronous%20Multiply,Accumulate%20Instruction%3A%20wgmma.wait_group).

On Blackwell (SM100+), warpgroup MMA continues as a supported feature, now managed under the **tcgen05** family. The 5th-gen Tensor Core ISA defines similar semantics using tcgen05.mma\_async and the general tcgen05.commit\_group/wait\_group for any async ops (including MMA). In practice, the programming model for large warpgroup MMAs on Blackwell is similar to Hopper’s but with unified pipeline controls. One still divides the CTA’s warps into groups of 4, uses *async* MMA instructions to overlap compute and data movement, and then synchronizes with minimal software intervention. The warpgroup concept (4 warps, base warp index multiple of 4\) remains the same[\[30\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=,397). Blackwell’s enhancements allow even more flexible shapes and the mixing of asynchronous copies (TMA) and compute in the same pipeline.

**Key points:** \- **Warpgroup \= 4 warps:** Hardware executes warpgroup MMA across 128 threads. Warps 0–3 form one warpgroup, 4–7 the next, etc., as long as the first warp ID is divisible by 4[\[30\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=,397). \- **Async execution:** wgmma.mma\_async allows issuing the multiply without immediate blocking. Threads can overlap global memory loads (via TMA) with matrix math. A sequence might load tiles with cp.async.bulk.tensor, issue several wgmma.mma\_async instructions, then use commit\_group and wait\_group to finalize[\[32\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=operation%20is%20performed%20in%20the,async%20proxy)[\[27\]](https://docs.nvidia.com/cuda/parallel-thread-execution/contents.html#:~:text=%2A%209.7.15.7.1.%20Asynchronous%20Multiply,Accumulate%20Instruction%3A%20wgmma.wait_group). \- **Shapes and types:** Hopper supported wgmma for many M=64, N up to 256, K up to 256 shapes in FP16/BF16/TF32/INT8. Blackwell extends this via tcgen05 but keeps similar shape support (e.g., .m64n256k16 etc. for FP16)[\[33\]](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-warpgroup#:~:text=Floating). The warpgroup ensures enough threads to handle these larger tiles. \- **Synchronization:** Minimal synchronization overhead – a single barrier or wait\_group can synchronize 128 threads’ worth of work, whereas previously one might need multiple syncs for smaller warp operations. This contributes to the **asynchronous pipeline model** used in FlashAttention and similar kernels on Hopper/Blackwell.

## CUDA C++ Programming Guide – Async Copies & TMA (CUDA 13.1, 2025\)

**Link:** *CUDA C++ Programming Guide – Asynchronous Data Copies* – [NVIDIA Docs](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html)

### Relevant sections

#### *TMA Descriptor Alignment and Layout Rules*

The **Tensor Memory Accelerator (TMA)** allows bulk asynchronous transfers of multi-dimensional arrays between global and shared memory. To use TMA for 2D+ tiles, one must create a **tensor map descriptor** describing the layout. This comes with specific **alignment and padding requirements**:

* **Global memory alignment:** Source addresses in global memory must be **128-byte aligned** for TMA transfers[\[34\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=,be%20aligned%20to%20128%20bytes)[\[35\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=,memory%20layout%20and%20access%20patterns). This generally means the starting address of your tensor (or sub-tensor) should be on a 128B boundary.

* **Shared memory alignment:** The destination shared-memory buffer should be aligned to the **swizzle pattern repeat length** (or simply 128 bytes to be safe)[\[34\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=,be%20aligned%20to%20128%20bytes). TMA can apply an optional bank-swizzle (32B, 64B, or 128B) to avoid bank conflicts. If the shared memory pointer isn’t aligned to the swizzle periodicity, an offset is introduced in how data is interleaved[\[36\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=,See%20comment%2C%20below)[\[37\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=Swizzle%20Pattern%20Pointer%20Offset%20Computation,apply%20the%20corresponding%20offset%20formula). For simplicity, aligning shared buffers to 128B (the max pattern) ensures no misalignment of the swizzle pattern.

* **Swizzle granularity:** The swizzle operates on 16-byte chunks. Shared memory is conceptually divided into segments (e.g. 32B, 64B, 128B) for bank rotation. If the inner dimension of the tile (the contiguous length per row in shared memory) exceeds the swizzle width, ensure the allocation covers the full swizzle span to avoid out-of-range access[\[38\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=,accommodate%20the%20full%20swizzle%20width)[\[39\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=CU_TENSOR_MAP_SWIZZLE_128B). Table 25 in the guide specifies that for swizzle modes 32B/64B/128B, the inner dimension must be ≤32/64/128 bytes respectively[\[40\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=CU_TENSOR_MAP_SWIZZLE_128B)[\[41\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=CU_TENSOR_MAP_SWIZZLE_64B).

* **Stride and shape constraints:** Each dimension of the global tensor must have a stride (in bytes) that is a multiple of **16 bytes** (the basic TMA transaction size)[\[42\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,be%20a%20multiple%20of%2032). If using 32B interleaving, strides must be multiples of 32B[\[43\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=2,be%20a%20multiple%20of%2032). Moreover, the total size in each dimension should be chosen so that each row’s length *in bytes* is a multiple of 16\. For example, a row of 3 int32 elements (12 bytes) is not allowed; padding to 4 elements (16 bytes) is required[\[44\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=and%2016%20bytes,found%20in%20the%20table%20below)[\[45\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=,be%20aligned%20to%20128%20bytes). The programming guide notes that a 4×3 matrix of 32-bit ints must pad each row to 16 bytes (adding 4 bytes of padding) to meet alignment[\[44\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=and%2016%20bytes,found%20in%20the%20table%20below).

* **Tensor map “box” dimensions:** The descriptor’s *boxDim* (the tile dimensions to copy) can be up to 256 in each dimension[\[46\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,a%20multiple%20of%2016%20bytes). All box dimensions must be \>0. If no element stride skipping is used, *boxDim* effectively equals the tile shape. If using element strides (to skip elements, e.g. strided copy), the guide explains that TMA will compute ceil(boxDim\[i\] / elementStride\[i\]) actual elements along that dimension[\[47\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,%60elementStrides%60%5Bi).

* **Interleaving modes:** TMA supports .interleave modes (e.g., **16B or 32B** interleaved channel formats) often used for images (NHWC layouts). When using CU\_TENSOR\_MAP\_INTERLEAVE\_NONE (no special channel interleaving) together with a swizzle, the “inner” contiguous dimension (in bytes) of the transfer must not exceed the swizzle segment size[\[48\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=TMA%20supports%20interleaved%20layouts%20like,32)[\[49\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,to%20be%20CU_TENSOR_MAP_INTERLEAVE_NONE). If it does, the swizzle would be applied in segments, which is unsupported – hence the requirement in the table that, e.g., for a 128B swizzle, the row length ≤128B (unless no swizzle or using interleaved mode to break it up).

* **Data type alignment:** The tensor map data type (e.g., FP16, INT8, etc.) also imposes alignment. For instance, **16-bit types** (FP16/BF16/half) require base addresses 16B-aligned (which is already covered by the 128B global align since 128B is divisible by 16). Special packed types like 4-bit and 6-bit (for sparse or compressed data) have unique rules: e.g., when using CU\_TENSOR\_MAP\_DATA\_TYPE\_16U4\_ALIGN16B, the first dimension’s size must be a multiple of 128 elements[\[50\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,be%203%2C%204%2C%20or%205)[\[51\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,be%20a%20multiple%20of%20128), and the base address must be 32B aligned[\[52\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=described%2C%20must%20be%2016%20byte,must%20be%2032%20byte%20aligned). These ensure that 4-bit values (packed into 16B blocks) line up correctly in memory[\[53\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=CU_TENSOR_MAP_DATA_TYPE_TFLOAT32_FTZ%20%20%20%20,CUtensorMapDataType).

* **Memory promotion and OOB fill:** The tensor map descriptor also includes flags for L2 cache promotion and out-of-bounds fill. These don’t affect layout per se, but note that enabling L2 promotion might require alignment to a cache line (128B), which is already satisfied by the above rules. Out-of-bounds elements can be set to zero or NaN via CUtensorMapFloatOOBfill if a tile partly spills outside the valid region[\[54\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=CU_TENSOR_MAP_DATA_TYPE_FLOAT64%2C%20%20%20%20,CUtensorMapDataType).

#### *mbarrier and Async Copy Coordination*

The CUDA Programming Guide emphasizes that asynchronous copies (whether LDGSTS or TMA) **must signal completion** before data is used. For TMA, there are two mechanisms: \- **Barrier completion:** When copying **to shared memory**, any thread can perform a \_\_syncthreads() after an mbarrier\_wait to ensure all data is ready[\[55\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=Asynchronicity,global%20or%20distributed%20shared%20memory)[\[56\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=signal%20that%20they%20have%20completed,and%20in%20the%20%2075). The guide notes that if TMA reads global→shared, you typically use a **shared memory barrier** or \_\_pipeline\_wait\_prior() (which internally uses an mbarrier) to wait for the copy[\[57\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=up%20to%20the%20hardware%20implementation,and%20in%20the%20PTX%20ISA). TMA writes (shared→global) are not globally visible to other threads, so only the launching thread can wait on its completion (via the bulk async-group mechanism or polling the barrier)[\[56\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=signal%20that%20they%20have%20completed,and%20in%20the%20%2075). \- **Bulk async-group:** For shared→global copies, the guide describes a “bulk async-group” completion: essentially the thread that did the copy will later call a PTX cp.async.bulk.commit\_group; cp.async.bulk.wait\_group; sequence (or the C++ API equivalent) to ensure all grouped copies are done[\[57\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=up%20to%20the%20hardware%20implementation,and%20in%20the%20PTX%20ISA). This doesn’t use an mbarrier but rather the built-in group tracking. Table 19 and 20 in the guide summarize which source→dest directions use which completion mechanism[\[58\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=Direction)[\[59\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=API). For example, global→shared uses a barrier, shared→global uses async-group[\[60\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=global)[\[59\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=API).

**Key takeaways:** \- Any asynchronous copy must be paired with either an **mbarrier wait** or a **group commit/wait** to avoid race conditions. The initiating thread should set up the barrier (including expected transaction counts if using TMA) and all consumer threads should wait on that barrier. \- The guide specifically notes that cuda::memcpy\_async (the high-level API) will automatically use TMA under the hood **if** both source and dest are 16-byte aligned and the size is a multiple of 16 bytes; otherwise it falls back to a regular copy[\[61\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=,endif%20%2F%2F%20__CUDA_MINIMUM_ARCH). Thus, aligning addresses and sizes to 16 bytes not only satisfies TMA requirements but also enables the faster path in the API. \- Best performance is achieved with **128B alignment** on both global and shared memory for these async copies[\[62\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=Source%20and%20destination,global%20memory%20is%20128%20bytes), and sizes that are multiples of 128B to fully utilize bus width. Misaligned or odd-sized copies may still work (if 16B aligned), but could incur additional overhead or cache pollution (e.g., 16B copies might hit L1). \- When using **swizzling** (to avoid bank conflicts in shared memory), ensure the shared memory layout matches the chosen swizzle mode. The programming guide provides formulas for computing the swizzle index offset if the shared base pointer isn’t aligned to the swizzle pattern[\[39\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=CU_TENSOR_MAP_SWIZZLE_128B)[\[63\]](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/async-copies.html#:~:text=%60%28reinterpret_cast%20%3Cuintptr_t%3E%28smem_ptr%29%2F128%29). In practice, aligning as recommended avoids having to deal with these offsets.

## CUDA Driver API – cuTensorMap (CUDA 13.1)

**Link:** *CUDA Driver API – Tensor Map Object Management* – [NVIDIA Docs](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html)

### Relevant sections

#### *cuTensorMapEncode\* API (Host-side Tensor Map creation)*

NVIDIA provides low-level APIs to create tensor map descriptors for TMA. These are part of the CUDA driver API (or via cudaTensorMap.h in the runtime). The key functions are:

* **CUresult cuTensorMapEncodeTiled(...)** – Create a tensor map for a tiled multi-dimensional region[\[64\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=CUresult%20%C2%A0%20147%20,CUtensorMapL2promotion%20%C2%A0l2Promotion%2C%20%20152%C2%A0oobFill)[\[65\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=Create%20a%20tensor%20map%20descriptor,CUtensorMapL2promotion%20%C2%A0l2Promotion%2C%20%20152%C2%A0oobFill). This is used for standard dense tensors (up to 5D) that don’t involve the specialized im2col mode.

* **CUresult cuTensorMapEncodeIm2col(...) / ...Im2colWide(...)** – Create tensor maps for image-im2col transformations[\[66\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=Functions). These include parameters for “pixel box” regions and are used in convolution lowering. (Im2colWide is a variant where the width dimension is handled specially for efficiency[\[67\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=CUresult%20%C2%A0%20153%20,object%20representing%20im2col%20memory%20region).)

* **CUresult cuTensorMapReplaceAddress(CUtensorMap \*tensorMap, void \*globalAddress)** – Modify an existing tensor map to point to a different base address[\[68\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=void,with%20an%20updated%20global%20address). This is handy if you allocate a descriptor once and then reuse it for many different buffers of the same shape by just updating the base pointer.

Each cuTensorMapEncode\* function fills a CUtensorMap (a tensor map descriptor object; its address must be **64-byte aligned**) with the layout of a tensor in memory. Key parameters include: \- **Data type and rank:** You must specify a CUtensorMapDataType (e.g., FP16, BF16, TF32, various packed 4-bit types) and the tensor’s rank (number of dimensions, **must be 3, 4, or 5** for `cuTensorMapEncodeTiled`)[\[50\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,be%203%2C%204%2C%20or%205). Ranks 1–2 are not supported by `cuTensorMapEncodeTiled` (1D copies use a simpler path). \- **Global address and strides:** The base address (globalAddress) of the data in GPU memory must be **16-byte aligned**[\[69\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,must%20be%2032%20byte%20aligned) (some types impose stricter alignment: e.g., 4-bit packed types require 32B alignment[\[52\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=described%2C%20must%20be%2016%20byte,must%20be%2032%20byte%20aligned)). You provide an array globalDim\[rank\] with the size of each dimension in elements[\[70\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,1%20dimensions), and globalStrides\[rank-1\] with the stride (in bytes) for each dimension except the last[\[71\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,1%20dimensions%20%60pixelBoxLowerCorner)[\[72\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,is%20CU_TENSOR_MAP_DATA_TYPE_16U6_ALIGN16B%20or). All strides must be multiples of 16 bytes and \< 2^40[\[73\]\[74\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,the%20strides%20must%20be%20a). If using 32B interleaving or 4-bit align16 types, strides must be multiples of 32[\[75\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=2,the%20strides%20must%20be%20a)[\[76\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,be%20a%20multiple%20of%2032). \- **Tile (box) dimensions:** You also specify the size of the sub-tensor to copy in each dimension via boxDim\[rank\][\[77\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=CUtensorMapDataType%20%C2%A0tensorDataType%2C%20cuuint32_t%C2%A0tensorRank%2C%20void,tensorMap). Each box dimension must be ≤ 256 and \> 0[\[46\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,a%20multiple%20of%2016%20bytes). For example, if you want to copy a 128×128 tile from a larger matrix, boxDim \= {128, 128}. If the tensor map covers the entire tensor, boxDim can equal globalDim (but still ≤256 each). \- **Element traversal strides:** Optionally, elementStrides\[rank\] can be provided to skip elements within each dimension (for strided sampling)[\[78\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=cuuint64_t,with%20an%20updated%20global%20address)[\[47\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,%60elementStrides%60%5Bi). Each element stride must be 1–8. If all are 1, the copy is dense. Non-1 strides mean TMA will copy fewer elements (every *n*th element along that axis), effectively allowing gather/scatter patterns. \- **Interleave and swizzle modes:** You pass CUtensorMapInterleave interleave and CUtensorMapSwizzle swizzle enums to control how data is interpreted: \- **Interleave:** NONE, 16B, or 32B[\[79\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=%E2%80%8E%20%20%20%20typedef,CUtensorMapInterleave). This is mainly for channel-packed layouts like NHWC – e.g., INTERLEAVE\_16B means 8-channel blocks are 16 bytes (for FP16), so likely used for C8 feature maps. \- **Swizzle:** NONE, 32B, 64B, 128B, or advanced 128B modes (with atomic 32B/64B segments)[\[80\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,CUtensorMapSwizzle%20which%20is%20defined%20as). Swizzling reorders how rows map onto shared memory banks to reduce conflict. Only certain combinations are allowed: e.g., if interleave \== 32B, then swizzle *must* be 32B[\[81\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=Data%20are%20organized%20in%20a,is). Also, some packed data types restrict swizzle (e.g., 6-bit data only supports NONE or 128B swizzle)[\[82\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=CU_TENSOR_MAP_INTERLEAVE_32B%2C%20,Load%20%26%20Store). \- **L2 promotion & OOB fill:** The l2Promotion parameter can hint that the accessed range should be fetched to L2 cache (choices are None, 128B, 256B, etc., corresponding to caching the first segments). The oobFill parameter controls out-of-bounds behavior: either zero-fill or NaN for elements beyond tensor boundaries[\[54\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=CU_TENSOR_MAP_DATA_TYPE_FLOAT64%2C%20%20%20%20,CUtensorMapDataType). These are optional but can be useful (for example, in convolutions to pad edges with zeros via hardware instead of software).

**Important constraints:**  
When calling cuTensorMapEncodeTiled, several **hard constraints** must be met or else the function will error: \- The CUtensorMap\* tensorMap output pointer must point to **64-byte aligned** memory[\[83\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,aligned%20to%2064%20bytes) (the descriptor object’s address must be aligned to 64B). \- The base globalAddress must be properly aligned (16B or 32B as noted)[\[69\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,must%20be%2032%20byte%20aligned). \- tensorRank must be 3, 4, or 5[\[50\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,be%203%2C%204%2C%20or%205). (If you have a 2D tensor, you still need rank=3 with one of the dimensions of size 1, for example.) \- Dimensions in globalDim must be ≤ 2^32-1 and \>0[\[84\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,be%20a%20multiple%20of%20128). If using 4-bit or 6-bit packed types, the first dimension (usually width) often must be a multiple of 128 to align groups of 128 elements into whole bytes[\[85\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=following%20requirements%20need%20to%20be,be%20a%20multiple%20of%20128). \- Strides in globalStrides must be ≥ the dimension length in bytes and satisfy the alignment (multiple of 16\)[\[73\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,the%20strides%20must%20be%20a)[\[86\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=%E2%80%8E%20%20%20%20globalStrides,%28globalDim%5Bi%5D). Typically you set globalStrides\[i\] \= globalStrides\[i-1\] \* globalDim\[i-1\] for a contiguous layout[\[87\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=%E2%80%8E%20%20%20%20globalStrides,%28globalDim%5Bi%5D), with padding added to meet alignment if needed. \- boxDim each ≤ 256 and if no interleave, the product of the first dimension and element size must be a multiple of 16 bytes[\[46\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,a%20multiple%20of%2016%20bytes)[\[88\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=256,a%20multiple%20of%2016%20bytes). (This reiterates that each row we copy should be 16B-aligned length.) \- If using interleaving or swizzle, additional constraints apply as mentioned (e.g., with no interleave and a swizzle, inner dimension ≤ swizzle size)[\[48\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=TMA%20supports%20interleaved%20layouts%20like,32). Also, some packed datatypes forbid certain swizzles or require interleave==NONE[\[49\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=,to%20be%20CU_TENSOR_MAP_INTERLEAVE_NONE)[\[82\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=CU_TENSOR_MAP_INTERLEAVE_32B%2C%20,Load%20%26%20Store). \- The API returns CUDA\_SUCCESS on success or CUDA\_ERROR\_INVALID\_VALUE if any parameter doesn’t meet requirements[\[89\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=)[\[90\]](https://docs.nvidia.com/cuda/cuda-driver-api/group__CUDA__TENSOR__MEMORY.html#:~:text=Tensor%20map%20objects%20are%20only,through%20CUDA%20APIs%20and%20PTX). It’s up to the programmer to ensure all these constraints; otherwise you must adjust your tensor (add padding, etc.) before encoding.

## CUTLASS/CuTe Examples (Hopper & Blackwell TMA Pipeline)

**Link:** *CUTLASS 3.1+ Examples – Pipelined TMA and mbarrier* (NVIDIA CUTLASS commit 637b159)

### Relevant sections

#### *SM90/SM100 GEMM Mainloop with TMA and mbarrier*

NVIDIA’s CUTLASS library (and its new CuTe DSL) demonstrates how to use TMA and mbarriers in CUDA kernels. Below is a simplified excerpt from a CUTLASS example that performs a matrix copy-add using a 2-stage pipeline on **H100/Blackwell** GPUs. One warp (the “producer” warp) issues the TMA load asynchronously and signals an mbarrier, while all warps wait on the barrier before consuming the data:

// In device kernel (128 threads per CTA, one warp does copy)  
int warp\_idx \= cutlass::canonical\_warp\_idx\_sync();  
bool lane0 \= cute::elect\_one\_sync();  // one thread elected for barrier ops

// Shared memory and barrier setup  
\_\_shared\_\_ uint64\_t mbar;             // mbarrier in shared memory  
volatile \_\_shared\_\_ float smem\_tile\[TILE\_M\]\[TILE\_N\];  // target tile in SMEM

// Producer warp (warp 0\) loads global tile \-\> shared memory via TMA  
if (warp\_idx \== 0 && lane0) {  
    mbarrier\_init(\&mbar, /\*arrival\_count=\*/1);                   // 1 producer  
    mbarrier\_arrive\_expect\_tx(\&mbar, bytes\_to\_copy);             // set tx-count  
    cute::copy(tma\_load.with((uint64\_t&)mbar), gmem\_tensor\_tile, smem\_tile); // async copy  
}  
\_\_syncthreads();  // ensure all warps reach barrier wait  
mbarrier\_wait(\&mbar, /\*phase=\*/0);  // block until copy complete (phase 0\)

// Now all warps can safely read smem\_tile...  
process\_tile(smem\_tile);

*(Illustrative code – actual CUTLASS names differ. This corresponds to steps in Colfax’s TMA tutorial [\[91\]](https://research.colfax-intl.com/tutorial-hopper-tma/#:~:text=1,initiate%20the%20TMA%20load%2C%20we)[\[92\]](https://research.colfax-intl.com/tutorial-hopper-tma/#:~:text=,object).)*

In this code, warp 0 uses mbarrier.init to initialize the barrier with an arrival count of 1 (itself)[\[91\]](https://research.colfax-intl.com/tutorial-hopper-tma/#:~:text=1,initiate%20the%20TMA%20load%2C%20we). It then calls mbarrier.arrive.expect\_tx to both signal its arrival and set the expected byte count for the async copy (the PTX behind this is mbarrier.arrive.expect\_tx.shared::cta.b64)[\[93\]](https://research.colfax-intl.com/tutorial-hopper-tma/#:~:text=match%20at%20L272%20count%20for,which%20we%20compute%20on%20line). The cute::copy with tma\_load.with((uint64\_t&)mbar) issues the cp.async.bulk.tensor load, embedding the barrier’s pointer so hardware will perform a complete\_tx on it when done[\[94\]](https://github.com/BBuf/how-to-optim-algorithm-in-cuda/blob/01a19bec9b67313af394920714111f8b9a21dad3/cutlass/cfx-article-src/tma/scale_tma_kernel.h#L124-L132). All other warps wait at mbarrier.wait(0), which corresponds to PTX mbarrier.try\_wait.parity.shared::cta.b64 phase=0 – a blocking wait for the barrier’s phase 0 to complete[\[95\]](https://research.colfax-intl.com/tutorial-hopper-tma/#:~:text=4,thread%200%20arriving%20at%20the). The \_\_syncthreads() is used to avoid a warp divergence hazard (only warp 0 arrived, but all warps execute the wait)[\[92\]](https://research.colfax-intl.com/tutorial-hopper-tma/#:~:text=,object). Once the barrier flips phase, the shared tile is full and visible to all threads.

**Key details:** \- **Pipeline stage division:** CUTLASS uses a producer-consumer model. Here the producer is warp 0 performing the global→shared copy, and all warps are consumers after the wait. This overlap allows the copy to happen concurrently with arithmetic in prior iterations of a loop (in a full GEMM, you’d double-buffer tiles and do MMA on one tile while loading the next). \- **Barrier init and reuse:** The barrier is in shared memory and reused each iteration. CUTLASS’s utility calls initialize\_barrier set the barrier to arrival count \= number of producers (1 in this case)[\[91\]](https://research.colfax-intl.com/tutorial-hopper-tma/#:~:text=1,initiate%20the%20TMA%20load%2C%20we). Each iteration, arrive\_and\_expect\_tx resets the tx-count for the bytes of that tile and signals arrival in one go[\[93\]](https://research.colfax-intl.com/tutorial-hopper-tma/#:~:text=match%20at%20L272%20count%20for,which%20we%20compute%20on%20line). After the wait, the barrier’s phase auto-increments, ready for the next tile’s use without reinit. \- **Compact PTX emitted:** The high-level C++ corresponds to a few PTX instructions: mbarrier.init.shared.b64, mbarrier.arrive.expect\_tx.shared.b64, the cp.async.bulk.tensor ... mbarrier::complete\_tx::bytes copy, and mbarrier.try\_wait.parity.b64[\[96\]](https://research.colfax-intl.com/tutorial-hopper-tma/#:~:text=mbarrier%2C%20and%20the%20invocation%20of,to%20resolve%20the%20thread%20divergence)[\[97\]](https://research.colfax-intl.com/tutorial-hopper-tma/#:~:text=,this%20is%20the%20first%20use). Notably, CUTLASS wraps these in C++ intrinsics and ensures the barrier address is 8B aligned and in the CTA’s shared memory (so .shared::cta scope). \- **Performance considerations:** This approach avoids a device-wide \_\_syncthreads() after the copy (which would serialize all warps every tile). Instead, the warps remain mostly independent, and only synchronize when absolutely necessary through the lightweight barrier. This improves GPU occupancy and throughput for large tiled operations like GEMMs[\[98\]](https://research.colfax-intl.com/tutorial-hopper-tma/#:~:text=Also%2C%20given%20that%20TMA%20is,needed%20for%20their%20practical%20use)[\[99\]](https://research.colfax-intl.com/tutorial-hopper-tma/#:~:text=TMA%20load%20copies%20data%20from,will%20explain%20the%20involved%20concepts).

#### *Using tcgen05 (UMMA pipeline on Blackwell)*

On Blackwell (SM10x), CUTLASS introduces **warp-specialized kernels** that leverage tcgen05 asynchronous operations and **warpgroup MMA**. For instance, a Blackwell kernel might use PipelineUmmaAsync (UMMA \= Unified Multi-Modal Accelerator, referring to Tensor Cores with new capabilities) instead of the older PipelineTmaAsync. In such a kernel, **128 threads (4 warps)** cooperate as a warpgroup to issue wgmma.mma\_async instructions (through CuTe’s cute::gemm abstractions). The pipeline still uses mbarriers, but now potentially two barriers: one for **producer/Full** stages and one for **consumer/Empty** stages, as Blackwell’s **2× warpgroup** scheduling allows multiple threadblocks to overlap (CUTLASS uses a “full barrier” and “empty barrier” pair)[\[100\]](https://raw.githubusercontent.com/NVIDIA/cutlass/main/include/cutlass/pipeline/sm100_pipeline.hpp#:~:text=typename%20Impl%3A%3AThreadCategory%3B%20using%20Params%20%3D,template). The general workflow: 1\. **Allocate Tensor Maps and barriers:** Each cluster of thread blocks allocates mbarriers in shared memory for synchronization. CUTLASS aligns these barriers and uses initialize\_barrier\_array\_pair\_aligned to set them up collectively[\[101\]](https://raw.githubusercontent.com/NVIDIA/cutlass/main/include/cutlass/pipeline/sm100_pipeline.hpp#:~:text=init_barriers%28SharedStorage%26%20storage%2C%20Params%20params%29%20,ClusterShape%20cluster_shape%2C%20dim3). 2\. **Issue TMA loads asynchronously:** Producer warp (or warpgroup leader) calls mbarrier.arrive\_expect\_tx and issues tcgen05.cp.async (through cute::copy) similar to above[\[94\]](https://github.com/BBuf/how-to-optim-algorithm-in-cuda/blob/01a19bec9b67313af394920714111f8b9a21dad3/cutlass/cfx-article-src/tma/scale_tma_kernel.h#L124-L132). On Blackwell, this may also involve a **multicast** if copying to multiple CTAs in a cluster (Hopper added .multicast TMA). 3\. **Compute with wgmma:** Once data is in shared memory, the warpgroup executes wgmma.mma\_async instructions. These accumulate partial results in registers while the next tile’s data is being loaded (overlap). 4\. **Commit and wait (tcgen05.wait):** CUTLASS issues wgmma.commit\_group after a batch of MMA instructions, then uses either an mbarrier.wait or the new tcgen05.wait\_group to pause until the group of MMAs is finished[\[27\]](https://docs.nvidia.com/cuda/parallel-thread-execution/contents.html#:~:text=%2A%209.7.15.7.1.%20Asynchronous%20Multiply,Accumulate%20Instruction%3A%20wgmma.wait_group). In CUTLASS’s CuTe DSL, this is abstracted but effectively ensures all math operations are done before proceeding (similar to how we waited for cp.async copies). 5\. **Store results via TMA:** The output tile can be written back with another TMA store (cute::copy with a store descriptor) followed by cp.async.bulk.commit\_group/wait\_group for the stores (or using a fence)[\[102\]](https://github.com/BBuf/how-to-optim-algorithm-in-cuda/blob/01a19bec9b67313af394920714111f8b9a21dad3/cutlass/cfx-article-src/tma/scale_tma_kernel.h#L146-L150). In the earlier example code, after scaling the tile, they call cute::copy(tmaStore, smem\_tile, gmem\_out\_tile); cute::tma\_store\_arrive(); … tma\_store\_wait() to asynchronously write results and then signal completion[\[103\]](https://github.com/BBuf/how-to-optim-algorithm-in-cuda/blob/01a19bec9b67313af394920714111f8b9a21dad3/cutlass/cfx-article-src/tma/scale_tma_kernel.h#L144-L149).

**Key differences in the SM100 pipeline:** \- **Warp specialization:** By using warpgroup (128 threads) and new TC instructions, Blackwell kernels achieve greater parallelism per tile. The programming model is more complex (multiple barrier pairs, masks for peer warps in cluster, etc. as seen in CUTLASS PipelineUmmaAsync implementation[\[104\]](https://raw.githubusercontent.com/NVIDIA/cutlass/main/include/cutlass/pipeline/sm100_pipeline.hpp#:~:text=CUTLASS_ASSERT%28params.consumer_arv_count%20%3E%200%20%26%26%20,position%20of%20peer%20SMs)[\[105\]](https://raw.githubusercontent.com/NVIDIA/cutlass/main/include/cutlass/pipeline/sm100_pipeline.hpp#:~:text=CUTLASS_DEVICE%20void%20init_masks,These%20operations%20can%20be%20explicity)), but these details are handled inside CUTLASS. \- **Minimal CPU involvement:** All these pipelines are set up in device code. Host-side just launches the kernel with the appropriate Params (which include pre-encoded tensor maps and aligned shared memory sizes). \- **Code example:** The *CUTLASS scaleTMAKernel* example (from CUTLASS’s cfx-article-src/tma/scale\_tma\_kernel.h) shows a full Blackwell pipeline using TMA load, compute, and TMA store. It initializes a shared mbarrier, uses mbarrier.arrive\_and\_expect\_tx before a copy(..., tma\_load.with(mbar)), waits on the barrier, does compute (scaling the tile), then uses an analogous cute::copy(tmaStore, …) for output followed by cute::tma\_store\_arrive() and (implicitly) waiting[\[106\]](https://github.com/BBuf/how-to-optim-algorithm-in-cuda/blob/01a19bec9b67313af394920714111f8b9a21dad3/cutlass/cfx-article-src/tma/scale_tma_kernel.h#L99-L108)[\[94\]](https://github.com/BBuf/how-to-optim-algorithm-in-cuda/blob/01a19bec9b67313af394920714111f8b9a21dad3/cutlass/cfx-article-src/tma/scale_tma_kernel.h#L124-L132)[\[103\]](https://github.com/BBuf/how-to-optim-algorithm-in-cuda/blob/01a19bec9b67313af394920714111f8b9a21dad3/cutlass/cfx-article-src/tma/scale_tma_kernel.h#L144-L149). The output store in Blackwell uses a **fence** instead of an mbarrier for completion (since global memory writes can be made visible by fence\_view\_async\_shared() and a threadblock cluster sync)[\[107\]](https://github.com/BBuf/how-to-optim-algorithm-in-cuda/blob/01a19bec9b67313af394920714111f8b9a21dad3/cutlass/cfx-article-src/tma/scale_tma_kernel.h#L136-L140)[\[108\]](https://github.com/BBuf/how-to-optim-algorithm-in-cuda/blob/01a19bec9b67313af394920714111f8b9a21dad3/cutlass/cfx-article-src/tma/scale_tma_kernel.h#L146-L149). This highlights that Blackwell provides multiple ways to ensure memory consistency (either via mbarrier or a combination of fences and cluster sync for stores).

## FlashAttention-4 (FlexAttention, CuTe DSL)

**Link:** *FlashAttention 4 – FlexAttn Implementation (Dao et al., 2024\)* – [GitHub: Dao-AILab/flash-attention](https://github.com/Dao-AILab/flash-attention)

### Relevant sections

#### *Upstream FlashAttention v4 – Input Layout & Requirements*

FlashAttention-4 (FA4) is implemented in CUDA using the CuTe (CUTLASS) DSL to leverage TMA and warpgroup Tensor Cores. It introduces a flexible “score\_mod” interface for variants of attention (ALiBi, T5 relative bias, etc.), but to achieve high performance it imposes **strict layout requirements** on inputs: \- **Data types:** Query, Key, and Value tensors must be either **FP16 or BF16** (mixed precision not allowed)[\[109\]](https://github.com/sgl-project/sglang/blob/60a230b1fda7a405ea08000eb1adad840aa8f081/sgl-kernel/python/sgl_kernel/_fa4_interface.py#L114-L119). All three must have the **same dtype** and be on the GPU (CUDA device)[\[109\]](https://github.com/sgl-project/sglang/blob/60a230b1fda7a405ea08000eb1adad840aa8f081/sgl-kernel/python/sgl_kernel/_fa4_interface.py#L114-L119)[\[110\]](https://github.com/sgl-project/sglang/blob/60a230b1fda7a405ea08000eb1adad840aa8f081/sgl-kernel/python/sgl_kernel/_fa4_interface.py#L130-L138). There is limited FP32 support (some routines cast to FP32 for accumulation, but inputs are expected in half precision). \- **Shape and contiguous layout:** For non-variable length mode, Q, K, V are 4D tensors with shape **(batch\_size, seqlen, num\_heads, head\_dim)**. They must be in a contiguous layout such that the **last dimension (head\_dim)** is contiguous in memory (stride of 1 element)[\[111\]](https://github.com/Dao-AILab/flash-attention/blob/58fe37fba6b07ac0aa6e88a94d68f8378c901028/flash_attn/cute/interface.py#L58-L65)[\[112\]](https://github.com/Dao-AILab/flash-attention/blob/58fe37fba6b07ac0aa6e88a94d68f8378c901028/flash_attn/cute/interface.py#L138-L141). In practice, this means if the tensor was created with shape (B, S, H, D) in PyTorch, it should not be permuted – otherwise .contiguous() is called internally to copy it (FA4 does maybe\_contiguous() on Q, K, V)[\[111\]](https://github.com/Dao-AILab/flash-attention/blob/58fe37fba6b07ac0aa6e88a94d68f8378c901028/flash_attn/cute/interface.py#L58-L65)[\[113\]](https://github.com/Dao-AILab/flash-attention/blob/58fe37fba6b07ac0aa6e88a94d68f8378c901028/flash_attn/cute/interface.py#L136-L144). Additionally, the output and any auxiliary tensors (like the “learned sink” vector for some variants) are required to be contiguous in memory for their last dimension as well[\[114\]](https://github.com/sgl-project/sglang/blob/60a230b1fda7a405ea08000eb1adad840aa8f081/sgl-kernel/python/sgl_kernel/_fa4_interface.py#L120-L128)[\[115\]](https://github.com/sgl-project/sglang/blob/60a230b1fda7a405ea08000eb1adad840aa8f081/sgl-kernel/python/sgl_kernel/_fa4_interface.py#L124-L132). \- **Head dimension multiples:** The attention head dimension (D) must be a multiple of the alignment requirements. FA4 specifically requires **head\_dim ≤ 256**, and for FP16/BF16 this D must be divisible by 8 (since 8 × 2-byte \= 16 bytes)[\[116\]](https://github.com/sgl-project/sglang/blob/60a230b1fda7a405ea08000eb1adad840aa8f081/sgl-kernel/python/sgl_kernel/_fa4_interface.py#L144-L149). In general, head\_dim \* sizeof(datatype) must be divisible by 16 bytes. So valid head dims are typically 32, 64, 96, 128, 160, 192, 224, 256, etc. (Common choices like 64, 128, 160, 192 are all OK). If this condition isn’t met, the code will assert and not run[\[116\]](https://github.com/sgl-project/sglang/blob/60a230b1fda7a405ea08000eb1adad840aa8f081/sgl-kernel/python/sgl_kernel/_fa4_interface.py#L144-L149). \- **Head count and grouping:** FA4 supports grouped-QKV attention (e.g., multi-query attention where num\_heads\_query \= k \* num\_heads\_key). If using grouped heads (GQA/MQA), the number of query heads must be divisible by the number of key heads. The code asserts num\_head % num\_head\_kv \== 0[\[117\]](https://github.com/sgl-project/sglang/blob/60a230b1fda7a405ea08000eb1adad840aa8f081/sgl-kernel/python/sgl_kernel/_fa4_interface.py#L140-L148). There is an optimization to “pack” grouped heads (to treat them as larger vectors for memory efficiency), but it is automatically disabled if the grouping would cause misalignment or if using certain features (see below). \- **Sequence lengths:** FA4 supports both fixed-length and variable-length sequences. In fixed-length mode, Q and K have shape (B, S, H, D). In variable-length mode, the inputs are given as flattened 2D tensors (total\_n\_queries × D, etc.) plus prefix arrays cu\_seqlens\_q/cu\_seqlens\_k of shape (B+1) to delineate sequences. Those prefix arrays must be **int32** type and contiguous[\[118\]](https://github.com/sgl-project/sglang/blob/60a230b1fda7a405ea08000eb1adad840aa8f081/sgl-kernel/python/sgl_kernel/_fa4_interface.py#L119-L127). If provided, FA4 will operate in a mode where it loops over segments of the input. Notably, certain combinations (like variable length with score\_mod) are not fully supported in the current version (e.g., no custom score\_mod in backward with varlen – see below). \- **Memory alignment:** Internally, FA4 uses from\_dlpack to import torch tensors into CuTe. It assumes an alignment of **16 bytes for Q, K, V, and output** (passing assumed\_align=16)[\[119\]](https://github.com/sgl-project/sglang/blob/60a230b1fda7a405ea08000eb1adad840aa8f081/sgl-kernel/python/sgl_kernel/_fa4_interface.py#L208-L216). This means it expects the pointer addresses of these tensors to be 16-byte aligned (which is usually true for PyTorch allocations of FP16/BF16 tensors due to page alignment). The LSE (log-sum-exp) tensor is created as FP32 and is imported with assumed\_align=4[\[120\]](https://github.com/sgl-project/sglang/blob/60a230b1fda7a405ea08000eb1adad840aa8f081/sgl-kernel/python/sgl_kernel/_fa4_interface.py#L213-L220). If any of these alignments were off, the kernel might still work but could underperform; however, PyTorch’s allocator virtually always aligns to 256 bytes for large allocations, so this is generally satisfied.

**Repacking and fallback triggers:**  
FA4 will **reallocate or transpose data** in a few situations to meet the above constraints: \- If the last dimension of Q, K, V is not contiguous (e.g., the tensor was stored as (B, H, S, D) or any non-standard stride), FA4 calls .contiguous() on it, incurring an extra copy[\[111\]](https://github.com/Dao-AILab/flash-attention/blob/58fe37fba6b07ac0aa6e88a94d68f8378c901028/flash_attn/cute/interface.py#L58-L65)[\[113\]](https://github.com/Dao-AILab/flash-attention/blob/58fe37fba6b07ac0aa6e88a94d68f8378c901028/flash_attn/cute/interface.py#L136-L144). This ensures the data is in (B,S,H,D) row-major order in memory. \- If the head dimension is not a multiple of the alignment (16 bytes), the code will throw an error (assert) rather than silently copy. The expectation is that model architectures choose a compatible head size. For example, a 40-dimensional head (which is not multiple of 8 bytes for FP16) would not work – one would need to pad it to Forty or use a different attention kernel not requiring TMA. \- If **grouped attention packing** would cause misalignment, FA4 disables the packing optimization. Specifically, for SM100 devices, if pack\_gqa=True but the grouping factor doesn’t divide 128 evenly, or if using variable-length inputs, it sets pack\_gqa \= False[\[121\]](https://github.com/sgl-project/sglang/blob/60a230b1fda7a405ea08000eb1adad840aa8f081/sgl-kernel/python/sgl_kernel/_fa4_interface.py#L264-L273). Packing is an optimization where multiple heads are loaded together via one TMA transaction. When disabled, each head is handled independently (a slight slowdown). This is done to ensure that each copy is 128B-aligned and that the internal tiling works with the hardware’s constraints. \- The code chooses **block sizes** (tile shapes) heuristically based on head dimension and sequence length. For example, on SM90 it might use a larger n\_block\_size if head\_dim=128 and not causal[\[122\]](https://github.com/sgl-project/sglang/blob/60a230b1fda7a405ea08000eb1adad840aa8f081/sgl-kernel/python/sgl_kernel/_fa4_interface.py#L260-L268). These heuristics don’t require user intervention but can affect performance. They don’t affect correctness, just how work is partitioned. \- Outputs (out and lse) are allocated if not provided. If provided by the user, they must match expected shape, dtype, device, and also be contiguous in memory[\[123\]](https://github.com/sgl-project/sglang/blob/60a230b1fda7a405ea08000eb1adad840aa8f081/sgl-kernel/python/sgl_kernel/_fa4_interface.py#L173-L181)[\[124\]](https://github.com/sgl-project/sglang/blob/60a230b1fda7a405ea08000eb1adad840aa8f081/sgl-kernel/python/sgl_kernel/_fa4_interface.py#L178-L186). A non-contiguous output would trigger an assert and likely a failure (since they don’t explicitly clone output).

In summary, upstream FA4 expects well-behaved, aligned, contiguous inputs. It makes trade-offs (like disabling certain optimizations) to handle edge cases, but if an input fundamentally violates requirements (e.g., wrong dtype or misaligned head size), it will not run. These constraints ensure that the kernel can utilize **Tensor Memory Accelerator (TMA)** for efficient memory transfer and **Tensor Cores** for compute without hitting alignment faults or mis-configured tensor maps.

#### *SCOPE’s CuTe-based FA4 Fork – KV-Bias Integration*

*(SCOPE is an internal codebase – this section is partly inferred; verify against the code paths under `SCOPE_KV_BIAS_BACKEND=fa4`.)*

SCOPE integrates FlashAttention-4 to implement attention with a **KV-cache attention bias** via FA4’s `score_mod` hook. Most upstream requirements remain; the bullets below are **upstream constraints** plus **integration notes** (some inferred — verify against the code paths).

**Key constraints (FA4 + `score_mod`):**
- Q, K, V must be half-precision CUDA tensors, shape-compatible, and contiguous in the last dimension[\[128\]](https://github.com/sgl-project/sglang/blob/60a230b1fda7a405ea08000eb1adad840aa8f081/sgl-kernel/python/sgl_kernel/_fa4_interface.py#L114-L123)[\[116\]](https://github.com/sgl-project/sglang/blob/60a230b1fda7a405ea08000eb1adad840aa8f081/sgl-kernel/python/sgl_kernel/_fa4_interface.py#L144-L149).
- Head dim (D) must align to the kernel’s alignment requirements (often D % 8 == 0 for FP16)[\[116\]](https://github.com/sgl-project/sglang/blob/60a230b1fda7a405ea08000eb1adad840aa8f081/sgl-kernel/python/sgl_kernel/_fa4_interface.py#L144-L149).
- `score_mod` should be pure/elementwise and fast; complex Python logic is not executed at runtime (it is lowered/JIT’d)[\[129\]](https://research.colfax-intl.com/a-users-guide-to-flexattention-in-flash-attention-cute-dsl/#:~:text=CuTe%20DSL%20Implementation)[\[130\]](https://research.colfax-intl.com/a-users-guide-to-flexattention-in-flash-attention-cute-dsl/#:~:text=bias_tensor%20%3D%20aux_tensors,0%5D%5D.to%28cutlass.Float32%29%20return%20tSrS_ssa%20%2B%20bias).
- If `score_mod` is enabled, FA4 requires a derivative function (`score_mod_bwd`) for backward[\[127\]](https://github.com/Dao-AILab/flash-attention/blob/58fe37fba6b07ac0aa6e88a94d68f8378c901028/flash_attn/cute/interface.py#L56-L64).

**KV-bias integration notes (inferred; verify):**
- Upstream FA4 only supports custom `score_mod` in backward on **SM100**, and not with variable lengths[\[125\]](https://github.com/Dao-AILab/flash-attention/blob/58fe37fba6b07ac0aa6e88a94d68f8378c901028/flash_attn/cute/interface.py#L62-L64). Treat KV-bias via `score_mod` as **SM100-only** for backward, and expect varlen restrictions.
- Some upstream wrappers disable `pack_gqa` when grouping/alignment constraints aren’t met (including varlen)[\[121\]](https://github.com/sgl-project/sglang/blob/60a230b1fda7a405ea08000eb1adad840aa8f081/sgl-kernel/python/sgl_kernel/_fa4_interface.py#L264-L273); a KV-bias path may also need to do this if bias indexing makes grouping awkward.
- FA4 disallows enabling softcap (built-in log2 scaling) simultaneously with a custom `score_mod`[\[126\]](https://github.com/Dao-AILab/flash-attention/blob/58fe37fba6b07ac0aa6e88a94d68f8378c901028/flash_attn/cute/interface.py#L24-L28)[\[127\]](https://github.com/Dao-AILab/flash-attention/blob/58fe37fba6b07ac0aa6e88a94d68f8378c901028/flash_attn/cute/interface.py#L56-L64). Ensure this path does not enable both at once.
- For additive biases, `score_mod_bwd` is typically the identity on the score gradient (plus optional bias-gradient accumulation if the bias is learnable).
- Guard this path behind env flags and fall back cleanly on unsupported configurations (rather than relying on undefined behavior).

In conclusion: a CuTe-based FA4 + `score_mod` KV-bias path can stay within upstream layout contracts (contiguous FP16/BF16 Q/K/V, proper alignment, etc.), but it inherits upstream limitations (notably SM100-only backward support for custom `score_mod`, and varlen restrictions). Treat it as an optional fast path with explicit fallbacks.
