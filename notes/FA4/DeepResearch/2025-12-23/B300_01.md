## Ground truth on B300

* **B300 is compute capability 10.3 (SM103).** ([NVIDIA Developer][1])
* **B200 is compute capability 10.0 (SM100).** ([NVIDIA Developer][1])

So “B200 and B300 flexibility” really means “10.0 and 10.3 flexibility”.

## Why Triton is failing right now on SM103

Your error is:

> `ptxas fatal: Value 'sm_103a' is not defined for option 'gpu-name'`

That is almost always “ptxas too old for this SM target”.

Two key facts line up perfectly with your situation:

1. **CUDA Toolkit 12.9 is where compiler target support for SM 10.3 was added** (`sm_103`, `sm_103f`, `sm_103a`). CUDA 12.8 does not claim that support in the feature archive. ([NVIDIA Docs][2])

2. **Triton (and PyTorch’s bundled Triton) often ships a bundled `ptxas` based on CUDA 12.8**, and you may need to override it to point at a newer toolkit’s `ptxas` via `TRITON_PTXAS_PATH`. ([GitHub][3])

This also matches the Triton project’s own tracking: **sm103 support in Triton 3.4 worked “as long as ptxas shipped with CUDA 12.9 or newer is used.”** ([GitHub][4])

So your “Kernel B Triton likely OK” assumption is only true after you fix which `ptxas` Triton is invoking.

## So how hard is “a rebuild”?

It depends what you mean by rebuild, but the shortest honest answer is:

* **Getting Triton-based kernels (Kernel B, RoPE v2, anything Inductor-Triton) to compile on B300 is mostly a toolchain rebuild/config issue, not a kernel rewrite.**
* **Getting FA4/CUTE working on B300 is a separate, higher-friction path because of CUTLASS/CUTE arch targeting and the `nvidia-cutlass-dsl` vs `torch._inductor` cutlass shadowing problem.**

### 1) “Rebuild” to make Triton work on B300

Difficulty: **low to medium**.

You do not need to touch your Kernel B code first. You need Triton to use a `ptxas` that understands `sm_103a`:

* Install a CUDA toolkit that includes SM103 in ptxas (CUDA 12.9+ per NVIDIA’s feature archive). ([NVIDIA Docs][2])
* Point Triton at it (override the bundled one):

```bash
export TRITON_PTXAS_PATH=/usr/local/cuda-12.9/bin/ptxas  # or CUDA 13.x path
uv run python scripts/triton_sdpa.py --kernel-b
```

Why this is the right direction: PyTorch/Triton packaging has explicitly documented this exact workaround pattern when the shipped ptxas is too old. ([GitHub][3])

If that fixes Kernel B, you’ve restored your “safe” backend ladder (Triton fallback works on SM103).

### 2) “Rebuild” to make FA4/CUTE score_mod work on B300

Difficulty: **medium to high**.

There are two independent requirements:

1. **Toolchain can compile for SM103**
   You either need CUDA 12.9+ toolchain support for `sm_103*` targets, or you must intentionally compile something that is guaranteed to run across 10.x. NVIDIA explicitly says 12.9 adds SM103 targets. ([NVIDIA Docs][2])

2. **Your kernels must not rely on SM100a-only forward-incompatible PTX**
   NVIDIA’s Blackwell compatibility guide calls out that PTX using architecture-conditional targets like `sm_100a/compute_100a` is *not* forward/backward compatible. ([NVIDIA Docs][5])
   That is relevant because your CUTE investigation suggests a lot of the stack is keyed around SM100a/SM100f.

So “rebuild FA4 for B300” is feasible, but it is not just a recompile. You may need to audit what gets emitted (cubin vs PTX, and which targets) and possibly add SM103-specific variants.

## The crucial nuance: you might not need SM103-specialized cubins for everything

NVIDIA’s compatibility guide says a cubin built for a compute capability runs on the same major and higher minor (so 10.0 cubin should run on 10.3). ([NVIDIA Docs][5])

And NVIDIA’s family-specific target explanation explicitly groups **10.0 and 10.3 into the `sm_100f` family**, and even notes `sm_100` and `sm_100f` cubin output are aliases for that family. ([NVIDIA Developer][6])

Practical implication:

* For **“make it run everywhere”** you can often target **SM100 family** and include PTX, rather than chasing SM103-specific builds.
* For **“peak perf on B300”**, you probably will want proper SM103 builds once your toolchain and dependencies stabilize.
