## Packet 1: FlashAttention (FA2 + FA4/CuTe) shape, stride, layout requirements

Links

```text
FlashAttention repo (README, entry points, usage): https://github.com/Dao-AILab/flash-attention
FlashAttention v2.1.0 release note (causal alignment detail): https://github.com/Dao-AILab/flash-attention/releases/tag/v2.1.0
```

Sources:

Section headings to read and extract (exact headings vary by version, but these concepts exist)

* Public Python API surface (the main attention entry points, plus “packed QKV” and “varlen” variants)
* Varlen API parameters and invariants (cu_seqlens semantics, max_seqlen semantics, q/k/v packing rules)
* Dtype and head-dim support matrix (what combinations are allowed vs. rejected)
* Causal masking convention details (this one has bitten people, and it did change at least once)

Key constraints and gotchas to explicitly pull into `layout-contracts.md`

1. **The exact accepted logical shapes** for Q/K/V in padded mode vs. varlen mode, and for packed-QKV variants (write them as strict contracts, not “usually”).
2. **Stride and contiguity requirements** (especially “head-dim contiguous” and whether Q/K/V must be contiguous overall, plus any allowed non-contiguous cases).
3. **Alignment rules** (base-pointer alignment expectations, and any alignment-related performance cliffs).
4. **Head-dim constraints**: allowed set, multiple-of constraints, and what happens outside the set (error, fallback, repack, or slower kernel).
5. **GQA/MQA head mapping rules**: how Hq and Hkv relate, and whether the kernel assumes a specific grouping or layout.
6. **Causal convention**: record the exact definition (top-left vs bottom-right alignment). FlashAttention explicitly notes a causal alignment convention change in v2.1.0, so hard-code the convention in your docs and tests.
7. **Varlen cu_seqlens dtype and device assumptions**: whether cu_seqlens must be int32, CPU vs GPU, monotonicity, last element equals total tokens, etc. (This is the kind of constraint that silently breaks “fused pack” attempts.)
8. **“What triggers internal repacking”**: list every case you observe that causes an internal transpose/repack (this is usually the hidden “layout_copies” time).
9. **Numerics modes**: deterministic flags, dropout behavior, softmax scaling expectations, and any constraints that change numerics (so you can set tolerances).
10. **Performance-sensitive layout preference**: even when multiple layouts are accepted, only some are “fast-path.” Encode that in the contract as “accepted vs fast.”

Why this packet matters for your fusion plan

* Your first safe fusion kernel is almost certainly a “post-projection pack” that targets the **fast-path** layout, not just any accepted layout. Without the FA fast-path layout documented, you can accidentally fuse yourself into a slower overall pipeline.

## Packet 2: CUTLASS/CuTe extension points for TMA pipelines and custom epilogues

Links

```text
CUTLASS pipeline overview (includes "Pipelines with TMA"): https://docs.nvidia.com/cutlass/latest/pipeline.html
CUTLASS docs index: https://docs.nvidia.com/cutlass/latest/index.html
```

Sources:

Section headings to read and extract

* Pipeline overview (what a pipeline stage is, where synchronization lives)
* “Pipelines with TMA” (how TMA integrates into the producer-consumer design)
* Where epilogues and “fusion” plug in (CUTLASS vocabulary for “what you can customize without rewriting the whole GEMM”)
* CuTe layout/tiling primitives (how layouts get represented so you can reason about transforms, not guess)

Key constraints and gotchas to extract into `blackwell-primitives-cheatsheet.md` and the kernel experiment ladder

1. **What “TMA pipeline” actually means in CUTLASS terms**: what the producer does, what the consumer does, and what the synchronization object is.
2. **Tile and alignment constraints** that TMA imposes on shared-memory tiles (this will bound your “pack kernel” tile shapes).
3. **Stage depth vs register pressure**: the pipeline depth that looks great on paper can destroy occupancy if registers explode. Document how you will measure and cap this.
4. **Warp specialization patterns**: how CUTLASS expects you to split work (copy roles vs compute roles) so you do not fight the library’s intended design.
5. **Where to fuse safely**: identify the intended epilogue extension points (bias, scale, activation, conversion) vs. places where “fusion” risks breaking the main MMA schedule.
6. **How to tell you are on the intended path**: kernel naming patterns, compile-time guards, and runtime logging so you can detect silent fallbacks.
7. **Integration principle**: if you can delete layout copies by changing a layout contract and using a supported epilogue/layout path, that is often lower risk than rewriting the core GEMM.
8. **What CUTLASS expects from your memory descriptors** (especially if you end up matching layouts to enable TMA-friendly loads).

## Packet 3: NVIDIA PTX ISA references for naming and semantics (cp.async.bulk*, mbarrier, WGMMA, tcgen05, TMEM)

Link (PDF)

```text
PTX ISA reference (use version matching your toolkit): https://docs.nvidia.com/cuda/parallel-thread-execution/index.html
```

Source:

Section headings to read and extract (in the PTX ISA that matches your CUDA toolkit)

* Asynchronous copy families (for cp.async and bulk variants)
* Barrier / synchronization instructions (including mbarrier semantics)
* Warp-group MMA instruction families (WGMMA-related semantics and constraints)
* Any tensor-memory or tensorcore pipeline instruction families relevant to SM10x

Key constraints and gotchas to extract into your cheatsheet (so you do not cargo-cult)

1. **Minimum `.target` / SM** required for each instruction family you name (so you can state “SM10x+ only” precisely).
2. **Operand alignment requirements** for bulk copy and tensor instructions (what must be aligned, and to what).
3. **Address space rules** (global vs shared vs local, and what the instruction actually expects).
4. **Memory ordering guarantees**: what a barrier guarantees and what it does not guarantee (this is the root of many “works in small tests, fails in training” bugs).
5. **How to recognize the same concept in PTX vs SASS**: what names you’ll see in disassembly vs what you’ll write in inline PTX (so your profiling notes stay coherent).
6. **Scope semantics**: CTA scope vs cluster scope vs warp scope, and how that changes correctness.
7. **Failure mode taxonomy**: the classic errors (misaligned bulk copies, incorrect barrier arrival/wait protocol, consumer reading stale tiles).

Practical note

* The archive PDF above is a canonical example of the PTX ISA style, but for Blackwell-era work you should always align your “naming and constraints” to the PTX ISA version that matches the toolkit you are compiling with.

## Packet 4: cuDNN frontend/backend API, Conv3d planning, and graph capture

Links

```text
cuDNN backend CNN library docs: https://docs.nvidia.com/deeplearning/cudnn/backend/latest/api/cudnn-cnn-library.html
cuDNN docs index: https://docs.nvidia.com/deeplearning/cudnn/
```

Source:

Section headings to read and extract

* Backend convolution graph building blocks (tensor descriptors, convolution descriptors, op graphs)
* Execution plans and engine configs (how you choose and cache the plan)
* Workspace planning (how much, when it is required, and how it interacts with plan selection)
* Guidance around capture and stability (what must be fixed if you want CUDA Graph capture to be reliable)

Key constraints and gotchas to extract into `vae-decode-architecture.md`

1. **Which tensor layouts and strides are supported** for Conv3d, and which layouts degrade kernel selection (this is the Conv3d -> Conv2d hunting ground).
2. **Which knobs drive algorithm selection** (workspace size, deterministic flags, math mode, dtype).
3. **Plan caching strategy**: what needs to be stable across runs to reuse plans without re-autotune.
4. **Graph capture constraints**: what must be preallocated and what must not change shape if you want capture to stick.
5. **Conv3d performance cliffs**: grouped conv, unusual strides, odd channel counts, and tiny T dimensions can trigger suboptimal paths.
6. **Rewrite candidates**: Conv3d with time-kernel = 1 and strideT = 1 is often a prime candidate for a Conv2d rewrite by folding T into batch (but only if it actually deletes work rather than adding reshapes).

## How this wires directly into your two High Priority internal docs

### Fill `layout-contracts.md` using Packets 1 + 2

* Packet 1 tells you: “What FA will accept and what is fast.” That defines the target layout you should produce after projection+RoPE+pack.
* Packet 2 tells you: “What constraints TMA pipelines and CUTLASS-style layouts impose,” which bounds what your pack kernel can efficiently do.

### Fill `other-in-self-breakdown.md` before writing any kernel

* If `layout_copies` are real and large, the first kernel is probably a pack kernel.
* If `rope_apply` dominates, fuse RoPE into pack.
* If KV-cache writes dominate, focus on cache layout and write path first.

(Those are “decision rules,” not assumptions. The point of the breakdown doc is to stop guessing.)

## Where I would push Level 6 research next (after the two High Priority docs)

These are “difficult but breakthrough-y” areas that tend to pay off, and also where people waste months if the contracts are not nailed down.

### 1) Delete layout copies without touching GEMM

This is the highest expected value early on.

* Make QKV projection output and FA input layout agree, or insert exactly one deliberate, tiled pack.
* If you can turn “3 to 5 tiny transforms” into “1 big predictable transform,” TMA + warp specialization becomes viable.

Pitfall

* If you fuse too aggressively and accidentally pull the GEMM away from a best-in-class cuBLAS kernel, your total step can get slower even if you deleted kernels. (Your experiment template should have an explicit “did we slow GEMM?” check.)

### 2) Fused RoPE + pack + KV-write as the first real Blackwell-style custom kernel

This is the first place TMA plus producer-consumer pipelines often make sense.

* Inputs are predictable tiles.
* Outputs are pure layout and store work (usually bandwidth-bound).
* You can pipeline “load next tile” while “compute RoPE on current tile” and “store previous.”

Pitfalls

* Barrier protocol mistakes, and register pressure from “doing everything” in one kernel.

### 3) Projection fusion only if you can do it at a real extension point

The “projection + epilogue does RoPE” dream is real, but only if you have a supported way to implement it without reimplementing GEMM scheduling.

* If CUTLASS epilogue extension points can express what you need, it is a credible Level 6 project.
* If not, it is closer to Level 7 risk.

### 4) VAE decode: Conv3d -> Conv2d hunting plus plan caching

This is where cuDNN planning and stability can pay off.

* If many Conv3d ops have time-kernel = 1, you may be paying Conv3d tax for a Conv2d problem.
* If shapes are stable, plan caching plus capture often matters as much as kernel choice.

If you want, the fastest “first execution” is:

1. Paste the templates from the zip into your repo (`notes/FA4/b300/…`).
2. Fill `layout-contracts.md` for exactly one representative shape end-to-end.
3. Do one stack-attributed `other_in_self` run and fill that table.
4. Only then pick between: pack-only kernel, RoPE+pack kernel, or KV-write kernel.

That sequence tends to prevent the classic failure mode: spending weeks on a gorgeous kernel that targets the wrong layout.
